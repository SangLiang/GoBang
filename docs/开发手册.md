# 开发手册


## 1. 进程与端口分工

- 静态游戏与前端打包：`npm start`（`gulp default`），默认访问 `http://localhost:5000`。
- 训练 API：`npm run server`（`node server/index.js`），默认 `http://127.0.0.1:3847`。
- 两者相互独立：游戏可在不启动训练 API 的情况下运行；训练接口失败不会中断对局。

## 2. 启动与进入模式流程

入口在 `public/js/main.js`：按键选择模式。

- `S`：进入单人模式 `SingleMode.start()`。
- `D`：进入双人模式 `DoubliePlayerMode.start()`。


## 3. 单人模式主循环

`public/js/SingleMode.js` 的关键行为：

1. 初始化状态：`hasLoggedSingleResult`、`moveCountSingle`、棋盘数组与回合变量。
2. 若 `NN_ASSIST_ENABLED=true` 且 `NN_LAMBDA!=0`，开局调用 `nnAssist.preloadWeightsFromTrainingApi()` 尝试拉取权重。
3. 玩家点击落子：
   - 写入 `gameList`；
   - 检查胜负，若玩家胜则调用 `trainingApi.appendTrainingLog({...result:"win-user"...})`；
   - 未结束则切换回合并调用 `ai.shotPiece(...)`。
4. AI 落子完成后再次检查胜负，若 AI 胜则写 `result:"win-ai"` 日志。

## 4. AI 如何做判断（核心）

入口在 `public/js/AI.js` 的 `shotPiece(gameTurn, gameList)`。

### 4.1 候选点生成

- 先执行 `gameLogic.checkDanger()`，得到 `window.needComputePlace`。
- **说明**（实现见 `public/js/gameLogic.js`）：
  - `checkDanger()`：遍历棋盘，对每个**空位**检查其 **8 邻域**（上下左右 + 四斜）是否至少有一颗**已有棋子（黑或白均可）**；若是，则把该空位 `{ x, y }` 加入列表。因此不是「只考虑对手旁边」——**己方进攻延伸线上、贴着己方子的空位**同样会进入候选。
  - `window.needComputePlace`：全局数组，**每次**调用 `checkDanger()` 时会先清空再重新填充；`AI.js` 只对这些点调用 `getTheGameWeight` 等逻辑。名称含 “Danger”，与注释「检测威胁点」一致，但当前实现是**候选落子集合**而非单独筛「威胁棋形」。
- **设计取舍与局限**：这是为性能做的**启发式剪枝**。五子棋里绝大多数有效着手都在已有子附近，该集合通常已覆盖主要攻守；但若盘面上存在**与所有棋子都不 8 邻接的空位**（例如子力极度分散时，某步想落在「远离所有子的空区」），当前逻辑**不会**给这些点算分，也就选不到它们——唯一例外是整盘尚无邻接候选时回退天元 `(7,7)`。若将来要支持「远距离好点」，需要扩展候选生成（例如扩大邻域半径、合并己方最长连线的延伸点、或在候选过少时并入全盘点等），与算力成本权衡。
- 若候选为空，回退下天元 `(7,7)`。

**示例：候选点如何得到、又如何被选中**
假设当前盘面只有三枚子：黑 `(7,7)`、白 `(7,8)`、黑 `(8,8)`。遍历空位时，凡是与这三枚子任一枚在 **8 邻域**内的空位都会加入候选，例如：`(6,6)`、`(6,7)`、`(6,8)`、`(7,6)`、`(8,7)`、`(8,9)`、`(9,9)` 等；而像 `(0,0)` 这种远离已有子的空位不会进入候选。去重后得到 `window.needComputePlace`（顺序取决于遍历顺序）。这一步只统计“哪些点值得算分”，不直接决定“下哪里”。

window.needComputePlace对应的数据结构示例：
```js
window.needComputePlace = [
  { x: 6, y: 6 },
  { x: 6, y: 7 },
  { x: 6, y: 8 },
  { x: 7, y: 6 },
  { x: 8, y: 7 },
  { x: 8, y: 9 },
  { x: 9, y: 9 }
];
```

接着 `AI.js` 只遍历 `window.needComputePlace`：对每个候选点先“假设在该点落子”，再调用 `getTheGameWeight` 分别评估**进攻价值**（己方在此落子的收益）和**防守价值**（拦截对手威胁的收益）。

遍历过程中会维护两组“当前最优”：
  - `attackBest = { point, score }`：到目前为止进攻分最高的点；
  - `defenseBest = { point, score }`：到目前为止防守分最高的点。
    每处理一个候选点，就用该点分值与当前最优比较，若更高则更新对应最优点。

```js
// 初始
attackBest = { point: null, score: -Infinity };
defenseBest = { point: null, score: -Infinity };

// 处理 { x: 6, y: 6 } -> attack=120, defense=80
attackBest = { point: { x: 6, y: 6 }, score: 120 };
defenseBest = { point: { x: 6, y: 6 }, score: 80 };

// 处理 { x: 8, y: 7 } -> attack=110, defense=160
// attack 不更新；defense 更新为更高分
defenseBest = { point: { x: 8, y: 7 }, score: 160 };

// 处理 { x: 8, y: 9 } -> attack=180, defense=70
// attack 更新为更高分；defense 不更新
attackBest = { point: { x: 8, y: 9 }, score: 180 };
```

若开启 NN（且 `lambda != 0`），则在规则分基础上为每个候选构造特征并计算 `assist`，形成混合分 `weight = patternScore + lambda * assist`；更新最优点时使用这个混合后的分值。最后直接比较候选点里的`attackBest.score` 与 `defenseBest.score`：谁更高就下谁，从而在“主动做势/进攻”与“优先补防/拦截”之间做一次最终决策。

沿用上面的棋盘与候选集，NN 混合评分示例（贴近当前实现量级）：
```js
lambda = 80;

// 注意：assist 来自 sigmoid 网络输出，通常在 (0,1)。
// 假设以下是“各候选在当前分支里的 patternScore 与 assist”
// { x: 8, y: 9 }: patternScore=180, assist=0.22 -> weight=180 + 80*0.22 = 197.6
// { x: 8, y: 7 }: patternScore=160, assist=0.52 -> weight=160 + 80*0.52 = 201.6
// { x: 6, y: 6 }: patternScore=120, assist=0.31 -> weight=120 + 80*0.31 = 144.8

// 混合后：
attackBest  = { point: { x: 8, y: 9 }, score: 197.6 };
defenseBest = { point: { x: 8, y: 7 }, score: 201.6 };

// 最终比较：201.6 > 197.6，因此本轮优先下 { x: 8, y: 7 }（防守优先）
```

对比两种算法之间的差异：
```js
// 不开 NN（或 lambda=0）时，比较的是纯 patternScore：
attackBest  = { point: { x: 8, y: 9 }, score: 180 }
defenseBest = { point: { x: 8, y: 7 }, score: 160 }
// 最终：180 > 160，选择 { x: 8, y: 9 }（进攻优先）

// 开启 NN（lambda=80）后，比较的是混合分：
attackBest  = { point: { x: 8, y: 9 }, score: 197.6 }
defenseBest = { point: { x: 8, y: 7 }, score: 201.6 }
// 最终：201.6 > 197.6，选择 { x: 8, y: 7 }（防守优先）
```

可见差异在于：NN 的 `assist` 会改变候选点之间的相对优势。在这个例子里，`(8,7)` 的 `assist` 更高，足以把原本略低的规则分“拉升”到第一，从而让最终决策由“偏进攻”切换为“先补防”。

前面我们看到最后的分数的结果很大程度上都要收到这个assist的值影响，这个assist到底是什么，来看看`assist` 的实现流程：

1. **为什么要有 `assist`**  
   纯规则分（`getTheGameWeight`）可解释、稳定，但在“分差接近”或“多点看起来都可下”的局面里，容易出现同分并列或策略单一。`assist` 的目的就是在不推翻规则分的前提下，提供一个可学习的辅助偏好，让 AI 在候选点之间做更细粒度排序。

2. **`assist` 怎么生成**  
   `assist` 是网络输出，不是网络内部参数。生成流程是：  
   - 先由 `checkDanger()` 得到候选点,通常是多个候选点；  
   - 对每个候选点计算规则分 `attackScore` / `defenseScore`；这个积分规则是查表得出的。基于本地的判断规则。  
   - 用 `nnFeatures.buildFeatures(...)` 构造 6 维输入（`nx`、`ny`、`attackNorm`、`defenseNorm`、`candCountNorm`、`moveProgress`），然后调用 `nnAssist.computeAssist(features)` 做前向计算，网络结构 `6 -> 4 -> 1`，激活函数为 sigmoid，输出 `assist`（通常在 `(0,1)`），这个会对每个候选点里的数据都执行一次，因为他们的6维输入的特征都不一样，所以最后输出的assist也是完全不同的。  

3. **`assist` 起什么作用**  
   `assist` 不单独决定落子，而是作为规则分的增益项参与混合：  
   - `bump = lambda * assist`  
   - `winWeight = attackScore + bump`  
   - `dangerWeight = defenseScore + bump`  
   然后分别取进攻/防守最大值再比较，决定最终落子。  
   因此，`assist` 的核心作用是“微调候选排序”：在接近局面中可能改判，在明显必杀/必防局面中通常不应抢主导。

### 4.2 两条评分分支

- 传统的基于纯规则分支（NN 关闭或 `lambda=0`）：
  - 对每个候选点计算进攻分与防守分（`getTheGameWeight`）；
  - 分别取最大威胁点后比较，选更高者。
- 神经网络辅助规则分支（NN 开启且 `lambda!=0`）：
  - 先算规则分；
  - 构造特征向量 `buildFeatures(...)`，在这构造前面我们提到的6个特征值；
  - 计算 `assist = nnAssist.computeAssist(features)`；
  - 混合评分：`weight = patternScore + lambda * assist`；
  - 同样比较“最大进攻点”与“最大防守点”来选最终落子。

```mermaid
flowchart TD
    A[AI.shotPiece] --> B[checkDanger\n生成候选点]
    B --> C{候选点为空?}
    C -->|是| C1[下 7,7]
    C -->|否| D{NN开启且lambda!=0?}

    D -->|否| E[纯规则评分\nattackScore defenseScore]
    D -->|是| F[buildFeatures]
    F --> G[computeAssist]
    G --> H[混合分\npattern + lambda*assist]

    E --> I[得到进攻/防守列表]
    H --> I
    I --> J[各自取最大威胁点]
    J --> K{winPoint > dangerPoint?}
    K -->|是| L[选进攻点]
    K -->|否| M[选防守点]
    L --> N[落子并判胜负]
    M --> N
```

### 4.3 评分量级观测方法

如果固定算法的评分量级差异过大，NN 辅助项可能被主评分“淹没”而失效。比如候选分中第一名为 15000，第二名为 1200 + NN 分；当 NN 分值较小时，第二名总分仍与第一名不在同一量级，NN 对排序几乎不起作用。

为判断“规则分与 NN 辅助项是否量级匹配”，可以进入棋局与AI对战10-20局，根据输出的采样日志，来分析指标。

- NN 分支日志标签：`[AI][NN][metric]`
- 纯规则分支日志标签：`[AI][RULE][metric]`

1) **前二分差占比（gapRatio）**
- 指标：`attackGapRatio` / `defenseGapRatio`（NN 分支对应 `...Mixed` 与 `...Pattern`）
- 单步定义：`gapRatio = (Top1 - Top2) / max(1, Top1)`（与代码实现一致，分母做 `max(1, Top1)` 防止除零）
- 含义：越大表示第一名相对第二名优势越大，排序越不容易被微小增益改变
- 经验判据：
  - 长期 `> 0.30`：规则分可能过陡，候选排序几乎固定
  - 常见 `0.05 ~ 0.20`：通常较健康
  - 长期 `< 0.03`：评分可能偏平，Top1/Top2 容易频繁互换（决策抖动）

2) **NN 增益占比（bumpRatio）**

这个指标可以理解为：NN 对这个候选点到底“插手了多少”。

- 指标：
  - `avgBump`：NN 平均绝对增益（单位仍是“分值”）
  - `avgBumpRatio`：NN 平均相对增益占比（可转成百分比）
- 单步定义（候选级）：
  - `bump = lambda * assist`
  - `bumpRatio = bump / max(1, patternScore)`
  - patternScore：规则算法中的分数
- 示例（单候选）：
  - 若 `lambda=120`、`assist=0.5`，则 `bump = 120 * 0.5 = 60`
  - 若该候选 `patternScore=12000`，则 `bumpRatio = 60 / 12000 = 0.005 = 0.5%`（NN 为轻微微调）
  - 若该候选 `patternScore=300`，则 `bumpRatio = 60 / 300 = 0.2 = 20%`（NN 影响偏强）
- 样本统计（一步内对候选取平均）：
  - `avgBump = mean(bump)`
  - `avgBumpRatio = mean(bumpRatio)`
- 经验判据（以下百分比均针对 `avgBumpRatio`）：
  - 多数 `< 0.1%`：NN 几乎不影响排序
  - 常见 `0.5% ~ 3%`：可在“接近分差”候选上产生有效微调
  - 频繁 `> 10%`：NN 可能过强，需防止压过规则战术

3) **是否改判（flipByNN）**
- 单步定义：`flipByNN = (selectedPatternOnly != selectedMixed)`，单条日志取值为 `true/false`
- 统计口径：对多步样本统计改判率 `flipRate = count(flipByNN=true) / 总步数`（可转成百分比）
- 含义：NN 是否真正改变了最终决策（不是只改变候选分数）
- 经验判据（以下百分比均针对 `flipRate`）：
  - `< 2%`：NN 基本无效
  - `5% ~ 20%`：通常是较合理的辅助强度
  - `> 35%`：NN 影响偏大，需关注稳定性

4) **分阶段观测（推荐）**
- 结合 `stonesOnBoard` 分桶看趋势：开局（`0~30`）/中盘（`31~90`）/后盘（`91+`）
- 一般期望：开局允许一定改判，后盘改判率下降（战术杀棋应更多由规则分主导）

结论判读示例：
- 若同时出现“`gapRatio` 偏高 + `avgBumpRatio` 很低 + `flipByNN` 接近 0”，通常说明**规则分本身可用，但与 NN 尺度不匹配，NN 辅助实际未生效**。

### 4.4 关于 `lambda` 参数如何去正确的修正调整

本节记录一次完整的 `NN_LAMBDA` 调参案例，作为后续同类问题的参考模板。

**调整原因**

- 现象：开启 NN 后，体感落子与纯规则 AI 几乎一致。
- 证据：观测日志里 `flipByNN` 长时间接近 0，`Mixed` 与 `Pattern` 的 gapRatio 基本重合。
- 根因：当前实现中 `assist` 来自 sigmoid 输出，范围约 `(0,1)`；混合项为 `bump = lambda * assist`。当 `lambda=0.08` 时，`bump` 只有小数级，难以影响规则分排序（规则分常在几十到上万量级）。

**调整过程**

1. 先补齐观测能力：在 `AI.js` 增加 `gapRatio`、`avgBumpRatio`、`flipByNN` 指标日志。  
2. 将日志持久化到服务端：新增 `PUT /api/log/append`，把每步指标写入 `data/log.log`（每行一条 JSON）。  
3. 基线采样（旧参数）：`NN_LAMBDA=0.08` 时，`avgBumpRatio` 约为万分位，`flipByNN` 近似 0，确认 NN 基本未生效。  
4. 参数调整：把 `config.js` 的 `NN_LAMBDA` 从 `0.08` 提升到 `80`，并同步更新注释建议区间。  
5. 再采样验证：观察 `data/log.log` 中同样指标，确认是否进入“可见但不过强”的区间。

**本次结果**

- 调整后（`lambda=80`）日志显示：
  - `avgBump` 稳定在约 `58~60`；
  - `avgBumpRatio` 进入可见区间（明显高于旧参数）；
  - 单局出现少量 `flipByNN=true`（例如约 `2/19`），说明 NN 已能在部分回合改变最终选点。
- 结论：相较旧参数，NN 已从“几乎无效”提升到“可参与决策”。

**经验总结**

- 不要只凭体感调参，必须先看三指标：`avgBumpRatio`、`flipByNN`、`gapRatio`。  
- 若出现“`avgBumpRatio` 极低 + `flipByNN≈0`”，优先考虑**量级失配**而非网络失效。  
- `lambda` 建议用阶梯法：基线 -> 中档 -> 高档（每档固定局数采样），避免一次拉满。  
- 目标状态通常是：NN 能在“接近分差”局面改判，但在明显杀棋/必防局面不抢主导。  
- 当前项目在此轮验证后，`lambda=80` 可作为后续实验起点；若后续出现过度改判，再回调到 `50~70` 区间复测。

## 5. 神经网络与训练基础（概念）

本节说明：在本项目里，**神经网络扮演什么角色**、**权重是什么**、**「训练」在本仓库里具体指哪种算法**，以及它与**规则 AI** 如何配合。

### 5.1 神经网络在做什么（推理）

- **直观理解**：把当前局面的一小段**数值信息**（**特征向量**，本项目里由 `buildFeatures` 得到）送进一个**固定结构的计算图**：每层对输入做**加权求和**，再经**激活函数**（非线性）得到下一层，最后输出一个（或多个）数。

与本项目网络形状一致的**数据流示意**（数字为各层向量长度；连线表示带权汇总，隐层在汇总后做激活）：

```mermaid
flowchart LR
  A["特征向量\n6 维"] --> B["隐层\n4 神经元\n加权求和+激活"]
  B --> C["输出\nassist 标量"]
```

- **图中「6 维」各表示什么**：网络**每一步评估**只针对**当前候选落子点** `(x,y)`，把这点的上下文压成 6 个数（顺序固定，见 §6 / `nnFeatures.js`）：
  1. **`nx`、`ny`**：落子在棋盘上的**位置**（列、行归一化到约 `[-1,1]`），让网络能区分角、边、中心等空间差异。  
  2. **`attackNorm`、`defenseNorm`**：该点在**规则 AI 眼里**的**进攻强度**与**防守强度**（由棋形分缩放、裁剪得到），等于把「规则已经算好的威胁信息」再交给 NN 做二次组合。  
  3. **`candCountNorm`**：当前局面**候选点大概有多少**（战场宽窄），帮助区分「局部纠缠」与「选择面较大」的局面。  
  4. **`moveProgress`**：**全局已下子比例**（开局 / 中盘 / 残局），让同一棋形在不同阶段可以有不同的辅助倾向。  
  合起来：6 维 = **在哪下、规则攻守多强、局面多挤、下到哪一阶段**，是「给 NN 的摘要」，不是棋盘全图。

- 图中「隐层 4 个神经元」是网络内部的 **4 个中间通道**：每个神经元都把 **6 维输入各自做一套加权求和，再经激活（非线性）**，得到 4 个中间值；这 4 个数再汇总成最后的 **`assist` 标量**。  
  - **意义**：在「输入信息」和「最终 assist」之间留一层可学习的**非线性组合**，容量比「6 维直接线性变 1 个数」略大，算力仍很小。  
  - **为什么是 4（不是 6、也不是别的数）**：隐层有几个神经元，**不是从特征维数推出来的公式结果**，而是实现时人为定的**网络规模**：神经元越多，**可调的连接权重越多**，理论上更能拟合复杂关系，但**每步算得更慢**、进化时**每个个体参数更多**、在「只靠对局胜负反馈」的设置里也**未必更好**。取 **4** 表示：在「几乎不占主流程算力、只当规则分上的小修正」这一前提下，给网络留一点**非线性组合能力**；若改成 2、8、16 等也可以，都属于换一套规模。**一旦改掉这个数字**，存下来的权重矩阵行列数全变，**旧权重文件就不能再直接加载**，所以代码里要用 **`nnAssistSchemaVersion` 等约定**让前端、进化脚本、JSON 里保存的形状**三方一致**，避免「结构是 6→8→1、文件里却是 6→4→1」这种 silently wrong。

- **权重（weights）**：各层之间的连接强度等参数，**存成数字**；**换一套权重 = 换一套「从特征到输出」的映射**。游戏里的 `GET /api/training` 拉取的主要就是这类参数 + 版本信息，用于在浏览器里**复现**与训练端一致的那张网。  
- **前向传播**：从输入算到输出、**不改动权重**的过程。对局里 AI 每次 `computeAssist` 只做前向传播，**不叫「训练」**。

### 5.2 「训练」在优化什么

- **目标**：找到一组权重，使网络在**你关心的任务**上表现更好（例如赢棋更多、得分更高）。  
- **常见路线（监督学习）**：有大量「输入 + 正确答案」样本，用**反向传播 + 梯度下降**逐步调权重。本项目**没有**把棋谱做成大规模标注数据集走这条主路径。  
- **本项目主路径（神经进化）**：把**整张网络的权重**当成「生物基因」，维护**种群**（多组不同权重）；让每个个体去下若干盘棋，用**胜负/得分**算**适应度**；**一代一代**做**选择、交叉、变异**，让平均适应度上升。不依赖棋步标签，依赖**与环境的对局结果**。  
- **与仓库脚本的关系**：`npm run evolve` 等在 Node 里跑的就是这类**离线进化**；产物是新的权重，再通过 `POST /api/training` 等方式写回训练仓库供游戏 `GET` 加载。

### 5.3 神经进化的评估过程（关键：一局一评分）

这是最容易混淆的概念，必须明确区分。

#### 不是"把200手作为输入训练"

你可能以为进化过程是这样：
```
第1手 → 网络预测 → 对比正确答案 → 调整权重
第2手 → 网络预测 → 对比正确答案 → 调整权重
...
第200手 → 网络预测 → 对比正确答案 → 调整权重
```

**这是监督学习的思路，本项目不是这样做的。**

#### 实际的评估流程

神经进化是"**一局一评分**"，局内只做推理，不更新权重：

```
┌─────────────────────────────────────────────────────┐
│  评估一个网络（如 evolved-xxx.json 里的那组权重）        │
├─────────────────────────────────────────────────────┤
│  for g = 0 to 3（默认GAMES_PER_INDIVIDUAL=4局）       │
│    ┌─────────────────────────────────────────────┐  │
│    │  下一局棋（100-200手）                       │  │
│    │    第1手: 局面A → 网络计算assist → 落子      │  │
│    │    第2手: 局面B → 网络计算assist → 落子      │  │
│    │    ...                                      │  │
│    │    第150手: 局面N → 网络计算assist → 落子     │  │
│    │    注意：整局内网络权重【完全不更新】          │  │
│    │                                             │  │
│    │  一局结束: 赢了 → 适应度 += 1（+速胜奖励）    │  │
│    │            输了 → 适应度 += -1                │  │
│    │            和了 → 适应度 += DRAW_FITNESS      │  │
│    └─────────────────────────────────────────────┘  │
│  end                                               │
│  返回：4局的平均适应度（如 0.5 表示2胜2负）          │
└─────────────────────────────────────────────────────┘
```

#### 关键区别对比

| 维度 | 监督学习 | 神经进化（本项目） |
|------|----------|-------------------|
| **训练信号** | 每一步都有"正确答案" | 只有最终胜负 |
| **权重更新** | 每步后立即调整 | 一局结束后只评分，**不更新** |
| **代际优化** | 单网络逐步改进 | 多网络竞争，好的被保留繁殖 |
| **学习方式** | 老师教对错 | 生存竞争，优胜劣汰 |

#### 种群评估示例

假设 `population=12`，每代评估如下：

| 个体 | 网络权重 | 评估过程 | 平均适应度 | 排名 |
|------|----------|----------|------------|------|
| 个体0 | 随机权重A | 下4局：1胜3负 | -0.5 | 差 |
| 个体1 | 随机权重B | 下4局：2胜2负 | 0.0 | 中 |
| 个体2 | 随机权重C | 下4局：3胜1负 | **0.8** | **好** |
| ... | ... | ... | ... | ... |

**下一代生成**（基于排名）：
- 精英保留（20%）：个体2直接复制到下一代
- 随机注入（10%）：全新随机权重，保持多样性
- 繁殖填充（70%）：个体2和个体1交叉变异产生新个体

#### 为什么这样设计？

1. **没有棋谱标签**：五子棋没有"每一步最优解"的数据集
2. ** credit assignment 困难**：很难判断第20手输棋和第120手输棋各负多大责任
3. **简单有效**：只告诉网络"这套权重下棋强不强"，让进化自己找到强的组合

这就是为什么叫"**神经进化**"而不是"**神经网络训练**"——它更像生物进化，没有老师，只有自然选择。

#### 神经进化的学习范式定位

神经进化**不属于无监督学习**，它属于**强化学习的一个分支**（也称为**进化计算/黑盒优化**）。

**三种学习范式对比**：

| 范式 | 训练信号 | 典型任务 | 本项目是否属于 |
|------|----------|----------|----------------|
| **监督学习** | 每一步都有"正确答案"（标签） | 图像分类、语音识别 | ❌ 不是 |
| **无监督学习** | 没有标签，发现数据内在结构 | 聚类、降维、密度估计 | ❌ 不是 |
| **强化学习** | 环境给出的奖励/惩罚信号 | 游戏AI、机器人控制 | ✅ **属于这类** |

**机器学习分类体系**：

```
机器学习
├── 监督学习（有标签）
├── 无监督学习（无标签，找结构）
└── 强化学习（有奖励信号）
    ├── 基于值函数（如Q-learning）
    ├── 基于策略梯度（如PPO）
    └── 进化方法 ──→ 本项目（神经进化）
```

**为什么不属于无监督学习**：

- **无监督学习**没有优化目标，只是"看看数据长什么样"（聚类、降维）
- **神经进化**有明确的**优化目标**（最大化赢棋奖励），只是不知道"正确答案"是什么，只知道"结果好不好"

**更准确的描述**：
1. **基于群体学习的强化学习** —— 用进化代替梯度下降
2. **黑盒优化** —— 把网络权重当参数，胜负当函数值，找最大值
3. **进化强化学习** —— 强化学习的一个子类

因此我们称之为"**神经进化**"最准确，避免与监督/无监督的常规分类混淆。

### 5.4 本项目中「规则 AI」与「NN 辅助」的分工

- **规则分**（`getTheGameWeight`）：基于棋形规则的标量，可解释、稳定。  
- **NN 输出**（`assist`）：对同一候选点的**额外标量微调**。  
- **混合**：`patternScore + λ * assist`。`λ=0` 时 NN 不参与，行为与纯规则一致；`λ` 过大时 NN 项会压过或扭曲规则分，可能出现**弱棋、不符合棋形直觉的着手**，一般宜从小 `λ` 逐步试，并在开启 NN 时仍把规则分作为主信号。

## 6. NN 输入特征与网络形状

定义在 `public/js/nnFeatures.js`。

- 特征维度：`FEATURE_DIM = 6`
- 隐层配置：`NN_ASSIST_HIDDEN = [4]`
- 网络形状：`[FEATURE_DIM, NN_ASSIST_HIDDEN, 1]`

### 概念速览

- **特征维度（`FEATURE_DIM`）**：网络**每次前向计算**喂进去的是一个**固定长度的数字向量**；**维度**就是向量里有几个数。每个数对应一类信息（落子位置、规则攻守分、候选点多少、棋局进度等），一起描述「当前要评估的**这一个**候选点」，而不是整盘棋的完整编码。维度越高，理论上能表达的信息越丰富，但计算与进化成本也越高；本项目用 **6** 维作**摘要**。

- **隐藏层（`NN_ASSIST_HIDDEN`）**：夹在**输入向量**和**最终输出**之间的**中间层**。每个神经元先把输入**加权求和**，再经**激活函数**做一步**非线性变换**——本项目 `Neuroevolution.js` 默认用 **Sigmoid**，把加权和压到约 `(0,1)`。本项目的 **`[4]`** 表示**一层**隐藏层、**4** 个神经元，比「6 维直接线性合成 1 个数」略灵活；**4** 是人为选取的网络规模，与 6 没有唯一公式对应，改宽度会改变权重矩阵形状，须与 `nnAssistSchemaVersion` 等约定一致，详见 5.1。

- **网络形状 `[6, [4], 1]`**：**6** 维特征 → **4** 个隐层神经元 → **1** 维标量输出（`assist`）。

### 隐藏层里的「4 个神经元」是什么

**一句话**：`buildFeatures` 得到 **6 个数**后，先送进 **4 个并排的小计算器**（每个各输出 1 个数），再交给 **最后 1 个**计算器合成 **`assist`**。**4** 指「中间通道有 4 条」，不是棋盘上的 4 个点。

**结构示意**（数据从左到右流过一次，就是一次 `compute(features)`）：

```mermaid
flowchart TB
  F["输入：features[0] … features[5]\n（6 维，无训练权重）"]
  H0["隐神经元 0：\nΣ(features[k]×w0[k]) → σ → h0"]
  H1["隐神经元 1：\nΣ(features[k]×w1[k]) → σ → h1"]
  H2["隐神经元 2：\nΣ(features[k]×w2[k]) → σ → h2"]
  H3["隐神经元 3：\nΣ(features[k]×w3[k]) → σ → h3"]
  O["输出神经元：\nΣ(hj×w_out[j]) → σ → assist"]
  F --> H0
  F --> H1
  F --> H2
  F --> H3
  H0 --> O
  H1 --> O
  H2 --> O
  H3 --> O
```

- **σ**：`Neuroevolution.js` 默认的 **Sigmoid** 激活，把加权和压到约 `(0,1)`。  
- **w0…w3**：每个隐神经元各有 **6** 个权重（连到 6 个输入）；**w_out** 有 **4** 个权重（连到 h0…h3）。

**和代码对象的对应**：

| 文档里说的 | 代码里 |
|------------|--------|
| 6 维输入 | `compute(features)` 的 `features`，先写入 `layers[0].neurons[0..5].value` |
| 4 个隐神经元 | `layers[1].neurons`，长度 **4**，每个 `Neuron` 有 **6** 个 `weights` |
| 1 个输出 | `layers[2].neurons[0]`，有 **4** 个 `weights`；`compute` 返回数组的第 0 项即 `assist` |

**建网过程**：`nnAssist.js` 把 `nnFeatures` 里的常数拼成 `[6, [4], 1]`，交给 `Neuroevolution`：

```javascript
// public/js/nnAssist.js（ensureNetwork 内，语义节选）
var ne = new Neuroevolution({
  network: [nnFeatures.FEATURE_DIM, nnFeatures.NN_ASSIST_HIDDEN, 1],
  // 即 [6, [4], 1]：输入 6、隐层 4、输出 1
  population: 1,
  elitism: 1,
  randomBehaviour: 0,
  mutationRate: 0,
  historic: 0
});
```

`neurons` 的数据结构（`Layer` / `Neuron` 定义）：

```javascript
// public/js/Neuroevolution.js（原始实现，节选）
// ---------------------------------------------
// Neuron：单个神经元对象
// - value：当前神经元“此刻”的输出值
//   - 输入层：由外部输入 features 直接写入
//   - 隐层/输出层：由「加权和 + 激活函数」计算得到
// - weights：当前神经元从“上一层”接入的所有边权重
//   - weights[k] 对应上一层第 k 个神经元 -> 当前神经元 的连接权重
var Neuron = function () {
	this.value = 0;   // 当前输出值
	this.weights = []; // 对上一层各输入的连接权重
}

// Layer：一整层神经元的容器
// - id：层索引（0=输入层，1=隐藏层，2=输出层）
// - neurons：本层神经元数组，元素类型都是 Neuron
var Layer = function (index) {
	this.id = index || 0;
	this.neurons = [];
}

// 按“层宽 + 输入维度”创建整层神经元
// nbNeurons：本层神经元个数（本项目输入层6、隐层4、输出层1）
// nbInputs：每个神经元接收的输入个数（等于上一层神经元个数）
Layer.prototype.populate = function (nbNeurons, nbInputs) {
	this.neurons = []; // 先清空，保证重复调用不会残留旧神经元
	for (var i = 0; i < nbNeurons; i++) {
		var n = new Neuron(); // 1) 新建第 i 个神经元，初始 value=0, weights=[]
		n.populate(nbInputs); // 2) 给该神经元生成 nbInputs 个随机权重（范围约 [-1,1]）
		this.neurons.push(n); // 3) 放入本层数组，成为 neurons[i]
	}
	// 结束后：neurons.length === nbNeurons，且每个 neurons[i].weights.length === nbInputs
}
```

这段实现可以按「对象层级」来理解：

1. `Neuron` 是**最小计算单元对象**  
   - `value`：这个神经元当前保存的输出值（输入层时相当于存放某一维特征；隐层/输出层时是激活后的结果）。  
   - `weights`：数组，`weights[k]` 表示「从上一层第 `k` 个神经元连到当前神经元」这条边的权重。

2. `Layer` 是**一组神经元的容器对象**  
   - `id`：层号。  
   - `neurons`：普通 JS 数组，元素类型是 `Neuron`。  
   - 所以 `this.layers[1].neurons[3]` 的含义是：「第 1 层里的第 4 个神经元对象」。

3. `Layer.prototype.populate(nbNeurons, nbInputs)` 在做什么  
   - 先 `this.neurons = []` 清空当前层；  
   - 循环 `nbNeurons` 次，每次 `new Neuron()` 新建一个神经元；  
   - 调 `n.populate(nbInputs)` 给这个神经元初始化 `nbInputs` 个随机权重；  
   - `push` 进 `neurons` 数组。  
   - 结果：本层会有 `nbNeurons` 个神经元，且每个神经元的 `weights.length === nbInputs`。

4. 对应到本项目的 `[6, [4], 1]`  
   - 输入层：`populate(6, 0)` => `neurons.length = 6`，每个神经元 `weights.length = 0`（输入层没有上一层）。  
   - 隐藏层：`populate(4, 6)` => `neurons.length = 4`，每个神经元 `weights.length = 6`。  
   - 输出层：`populate(1, 4)` => `neurons.length = 1`，该神经元 `weights.length = 4`。

5. 当前项目的“具体代码 + 具体数据”示例  
   下面这组输入直接对应 `nnFeatures.buildFeatures` 的 6 维格式（`[nx, ny, attackNorm, defenseNorm, candCountNorm, moveProgress]`）：

```javascript
// 例子：某候选点原始坐标为 (x=8, y=5)
// nx=(8-7)/7=1/7≈0.142857, ny=(5-7)/7=-2/7≈-0.285714
// 其余特征：attackNorm=0.80, defenseNorm=0.35, candCountNorm=0.24, moveProgress=0.40
var features = [0.142857, -0.285714, 0.80, 0.35, 0.24, 0.40];

// 本项目建网（nnAssist.js）
var ne = new Neuroevolution({
  network: [6, [4], 1]
});
var net = ne.nextGeneration()[0];
```

   这时 `net.layers` 的“形状”就是：

```javascript
// 只展示结构，不展开全部权重数值
net.layers.length === 3;               // 输入层/隐藏层/输出层
net.layers[0].neurons.length === 6;    // 输入层 6 个神经元
net.layers[1].neurons.length === 4;    // 隐藏层 4 个神经元
net.layers[2].neurons.length === 1;    // 输出层 1 个神经元

net.layers[0].neurons[0].weights.length === 0; // 输入层无上一层
net.layers[1].neurons[0].weights.length === 6; // 每个隐神经元接 6 个输入
net.layers[2].neurons[0].weights.length === 4; // 输出神经元接 4 个隐层输出
```

   一次前向传播后（`var out = net.compute(features)`）可理解为：

```javascript
// 第 1 步：把 6 维输入写入输入层 value
layers[0].neurons[i].value = features[i]

// 第 2 步：算 4 个隐层输出（每个都是 6 项加权和后过 Sigmoid）
h0 = sigmoid(f0*w00 + f1*w01 + ... + f5*w05)
h1 = sigmoid(f0*w10 + f1*w11 + ... + f5*w15)
h2 = sigmoid(f0*w20 + f1*w21 + ... + f5*w25)
h3 = sigmoid(f0*w30 + f1*w31 + ... + f5*w35)

// 第 3 步：输出层把 h0..h3 再做一次加权和并过 Sigmoid
assist = sigmoid(h0*wo0 + h1*wo1 + h2*wo2 + h3*wo3)

// 返回值
out.length === 1，out[0] 就是 assist
```

   参数总数也能直接数出来：`4*6 + 1*4 = 28`，这正是训练 JSON 中 `weights` 应该匹配的长度。

可把它想成一个三层嵌套结构：

```javascript
network.layers[layerIndex].neurons[neuronIndex] = {
  value: Number,
  weights: Number[] // 与上一层神经元一一对应
}
```

一句话理解：`this.layers[x].neurons` 就是一个数组，数组每个元素都是 `Neuron` 对象；输入层只写 `value`，隐层/输出层同时使用 `value + weights`。

前向传播（`Network.prototype.compute` 原始实现）：

```javascript
// public/js/Neuroevolution.js（原始实现，节选）
Network.prototype.compute = function (inputs) {
	// 将输入写入输入层各神经元的 value
	for (var i in inputs) {
		if (this.layers[0] && this.layers[0].neurons[i]) {
			this.layers[0].neurons[i].value = inputs[i];
		}
	}

	var prevLayer = this.layers[0]; // 上一层的输出作为当前层输入
	for (var i = 1; i < this.layers.length; i++) {
		for (var j in this.layers[i].neurons) {
			var sum = 0;
			for (var k in prevLayer.neurons) {
				// 当前神经元对上一层所有输出的加权和
				sum += prevLayer.neurons[k].value *
					this.layers[i].neurons[j].weights[k];
			}
			// 经激活函数得到当前神经元输出
			this.layers[i].neurons[j].value = self.options.activation(sum);
		}
		prevLayer = this.layers[i];
	}

	// 收集输出层所有神经元的 value 作为网络输出
	var out = [];
	var lastLayer = this.layers[this.layers.length - 1];
	for (var i in lastLayer.neurons) {
		out.push(lastLayer.neurons[i].value);
	}
	return out;
}
```

**权重个数（和 JSON 对不上时加载会失败）**：隐层 **4×6 = 24**，输出 **1×4 = 4**，共 **28** 个浮点数；须与 `nnAssistWeights.neurons` 为 `[6, 4, 1]` 且 `weights.length === 28` 一致，否则 `nnAssist.js` 里 `validateSaveShape` / `setSave` 不会采用。

**小结**：**4** 是人为定的**隐层宽度**（网络容量）：比「6 维直接接到 1 个输出」多一层可进化的中间组合，但层仍很窄，适合只做规则分上的轻量辅助。

### 当前 6 维特征：

补充：这里的“归一化”指把原始值压到更统一的范围（常见是 `[-1,1]` 或 `[0,1]`），减少不同量纲之间的尺度差异，便于网络稳定计算。  
以 `nx` 为例：`nx = (x - 7) / 7`，所以 `x=0 -> nx=-1`，`x=7 -> nx=0`，`x=14 -> nx=1`；`ny` 同理。
`clamp` 是限幅函数：`clamp(v, lo, hi)`。若 `v < lo` 返回 `lo`；若 `v > hi` 返回 `hi`；否则返回 `v`。  
实现代码（与 `public/js/nnFeatures.js` 一致）：

```javascript
function clamp(v, lo, hi) {
  if (v < lo) return lo;
  if (v > hi) return hi;
  return v;
}
```

实现原理：本质是一个三分支比较器，先判断是否低于下界，再判断是否高于上界，最后返回区间内原值；时间复杂度 `O(1)`、无副作用。  
等价数学表达：`clamp(v, lo, hi) = min(max(v, lo), hi)`，先把值抬到不小于 `lo`，再压到不大于 `hi`。
在本项目里它是归一化后的“安全护栏”：避免极端分数或异常值把输入拉出预期区间，提升 NN 计算稳定性。

1. `nx`：列坐标归一化。  
   公式：`nx = (x - 7) / 7`，`x ∈ [0,14]` 时 `nx ∈ [-1,1]`。
2. `ny`：行坐标归一化。  
   公式：`ny = (y - 7) / 7`，`y ∈ [0,14]` 时 `ny ∈ [-1,1]`。
3. `attackNorm`：进攻分归一化。  
   公式：`attackNorm = clamp(attackScore / 1e6, -1, 1)`。
   说明：当前实现里 `attackScore` 基本非负，所以实际值通常落在 `[0,1]`；`-1` 下界主要是为后续可正可负的评分特征预留（例如引入惩罚项或差分特征时可直接兼容）。
4. `defenseNorm`：防守分归一化。  
   公式：`defenseNorm = clamp(defenseScore / 1e6, -1, 1)`。
   说明：当前实现里 `defenseScore` 也基本非负，实际多为 `[0,1]`；保留 `-1` 下界属于通用对称区间设计，和 `nx/ny` 的 `[-1,1]` 习惯保持一致。
5. `candCountNorm`：候选点规模归一化。  
   公式：`candCountNorm = clamp(candidateCount / 50, 0, 1)`。
6. `moveProgress`：棋局进度（已下子数 / 225）。  
   公式：`moveProgress = clamp(stonesOnBoard / 225, 0, 1)`。

说明：`nnAssist.js`、`scripts/evolve-ai.js`、`scripts/verify-nn-lambda-effect.js` 已统一引用这组定义，避免魔数漂移。

## 7. 训练数据与权重读写

前端封装在 `public/js/trainingApi.js`：

- `GET /api/training`：从训练仓库读取 JSON（游戏侧主要用来取 **NN 辅助权重** 及配套版本信息）。
- `PUT /api/training/append`：追加单局日志 entry。
- 任何请求失败都返回安全值，不阻断对局。

三个接口对应同一份「训练仓库」`data/ai-training.json` 的三种业务动作（训练服务与游戏页面分进程/分端口运行时，通过它们读写这份仓库）：

### `GET /api/training`（读取 NN 权重等配置）

- **业务含义**：从训练仓库**只读**取出当前保存的内容。对游戏而言，**主要用途就是拿「神经网络辅助落子」要用的数据**——也就是**权重**（以及例如 `nnAssistSchemaVersion` 这类**判断是否可加载、是否与当前网络结构匹配**的配套信息）。  
- **说明**：接口返回的是**整份仓库 JSON**，里面往往还会带上对局流水等其它字段；**和 NN 直接相关、游戏会去用的，就是其中的权重与版本类字段**，其余字段是否参与逻辑由产品决定。  
- **若没有仓库或暂不拉取**：视为**没有可用的远程权重**，NN 辅助仍可用内置默认网；**拉取失败**时同样不强行依赖远程权重，对局继续。

### `POST /api/training`（整表替换）

- **业务含义**：用你提供的一份 JSON，**整体替换**仓库里的快照——相当于**发布新配置**、**整包导入**或**整表重置**。
- **典型场景**：离线进化或整理完成后，**一次性写入**新权重与版本，以及你希望文件根上出现的所有字段。
- **业务注意**：这是**整份替换，不是与旧文件合并**。新快照里没带的根级信息（例如之前的对局流水、其它元数据），在业务上视为**随旧快照一起被取代**。若既要更新权重又要保留日志，应在业务侧先拼好**完整**快照再 `POST`，对局流水仍建议用下面的「追加」接口持续积累。

### `PUT /api/training/append`（追加一条流水）

- **业务含义**：在仓库的 **`records` 流水**里**多记一条**（如单局结果：胜负、手数、时间等），**不要求**每次提交整份仓库。
- **典型场景**：每打完一局自动上报一条，供后续统计、再训练或复盘；与「整表替换」分工明确，避免每局都走全量发布。
- **业务注意**：每条会附带服务端记录的时间，与业务方传入的 `entry` 字段一起形成一条流水；一般不用于「改权重」——权重仍以 `GET` 读取、`POST` 整表发布为主。

## 10. 进化生成的权重文件（`evolved-*.json`）

### 10.1 文件生成方式

`scripts/evolve-ai.js` 是离线进化脚本，通过神经进化算法优化 NN 辅助网络的权重。

**运行方式**：
```bash
cd scripts
node evolve-ai.js
```

**可配置环境变量**：
- `POPULATION`（默认12）：每代种群大小
- `GENERATIONS`（默认5）：进化代数
- `GAMES_PER_INDIVIDUAL`（默认4）：每个个体评估时的对局数
- `NN_LAMBDA`（默认0.08）：评估时混合系数
- `DRAW_FITNESS`（默认-0.1）：和棋适应度
- `WIN_SPEED_BONUS`（默认0.04）：速胜奖励
- `FITNESS_NOISE`（默认0.015）：排序噪声

**文件生成逻辑**（`evolve-ai.js` 第212-227行）：
- 每次运行生成一个新文件
- 文件名使用 ISO 时间戳：`evolved-YYYY-MM-DDTHH-mm-ss-sssZ.json`
- 保存的是**整个进化过程结束后的最优权重**，不是某一代的中间结果

因此，多个 `evolved-*.json` 文件代表**多次独立的进化运行**，不是代数关系。

### 10.2 单次运行内的代际进化（代数之间如何继承）

在一次 `evolve-ai.js` 运行内部（例如 `GENERATIONS=5`），**代与代之间是继承进化的**，不是每次都从零开始。

#### 代际流转过程

```javascript
// evolve-ai.js 循环逻辑
for (gen = 0; gen < GENERATIONS; gen++) {
    nets = ne.nextGeneration();  // 获取下一代网络
    // ... 评估每个网络的适应度 ...
}
```

| 代数 | 种群来源 | 说明 |
|------|----------|------|
| **第0代** | 完全随机生成 | `firstGeneration()` 创建随机权重 |
| **第1代及以后** | 基于上一代进化 | `nextGeneration()` 通过选择、交叉、变异产生 |

#### 下一代的生成策略（由 `Neuroevolution.js` 控制）

假设 `population=20`，下一代的**20个个体**按以下方式生成：

##### 1. 精英保留（elitism=20%）

约 **4个** 最好的个体**原样复制**到下一代：

```javascript
// Neuroevolution.js 第351-357行
for (var i = 0; i < Math.round(0.2 * 20); i++) {  // i = 0,1,2,3
    if (nexts.length < 20) {
        nexts.push(JSON.parse(JSON.stringify(this.genomes[i].network)));
        // genomes[0]是第1名，genomes[1]是第2名... 直接深拷贝，权重一点不变
    }
}
```

**作用**：保留优秀基因不丢失，确保最好的解不会退化。

##### 2. 随机注入（randomBehaviour=10%）

约 **2个** 个体使用**最优结构但权重全部重新随机**：

```javascript
// Neuroevolution.js 第359-369行
for (var i = 0; i < Math.round(0.1 * 20); i++) {  // i = 0,1
    var n = JSON.parse(JSON.stringify(this.genomes[0].network));  // 复制第1名结构
    for (var k in n.weights) {
        n.weights[k] = randomClamped();  // 重新随机 [-1,1]
    }
    if (nexts.length < 20) {
        nexts.push(n);
    }
}
```

**作用**：保持种群多样性，防止过早收敛到局部最优。

**形象比喻**：像往种群中"空投"几个全新个体，避免大家挤在一个小圈子里。

##### 3. 交叉变异繁殖（剩余约70%）

约 **14个** 个体通过**交叉（crossover）**和**变异（mutation）**产生：

```javascript
// Neuroevolution.js 第371-388行
var max = 0;
while (true) {
    for (var i = 0; i < max; i++) {
        // 按适应度加权配对：i越小（排名高）和 max 配对产生子代
        var childs = this.breed(this.genomes[i], this.genomes[max], 1);
        for (var c in childs) {
            nexts.push(childs[c].network);
            if (nexts.length >= 20) {  // 达到20个就停止
                return nexts;
            }
        }
    }
    max++;
    if (max >= this.genomes.length - 1) {
        max = 0;  // 循环使用
    }
}
```

**交叉过程（breed函数）**：

```javascript
// 两个父代各贡献一部分权重
for (var i = 0; i < nbNeurons; i++) {
    for (var k in this.genomes[i].network.weights) {
        if (Math.random() <= 0.5) {
            // 50%概率取父代1的权重
            data.network.weights.push(this.genomes[i].network.weights[k]);
        } else {
            // 50%概率取父代2的权重
            data.network.weights.push(this.genomes[max].network.weights[k]);
        }
    }
}
```

**变异过程（在breed内）**：

```javascript
// Neuroevolution.js 第329-337行
for (var i in data.network.weights) {
    if (Math.random() <= 0.15) {  // 15%概率变异
        data.network.weights[i] += Math.random() * 0.5 * 2 - 0.5;  
        // 加上 [-0.5, 0.5] 的随机扰动
    }
}
```

**作用**：在父母基础上微调，渐进优化，让种群向更优方向进化。

##### 数量汇总（以population=20为例）

| 来源 | 比例 | 数量 | 生成方式 |
|------|------|------|----------|
| **精英保留** | 20% | **4个** | 直接复制，权重不变 |
| **随机注入** | 10% | **2个** | 结构复制，权重重新随机 |
| **交叉变异** | 70% | **14个** | 父母交叉后部分权重变异 |
| **总计** | 100% | **20个** | 数量保持恒定 |

**关键特性**：
- 种群数量**始终保持20个**，不多不少
- 子代质量总体趋向提升，因为高分个体更常被选为父代
- 随机注入和变异保持多样性，防止过早收敛

#### 多次独立运行的关系

**关键点**：每次运行 `node evolve-ai.js` 都是完全独立的：

```
第1次运行：随机种群 → 进化5代 → 最优权重 → evolved-09-00-00.json
第2次运行：随机种群 → 进化5代 → 最优权重 → evolved-09-30-00.json（从头开始！）
第3次运行：随机种群 → 进化5代 → 最优权重 → evolved-10-00-00.json（从头开始！）
```

- 8 个 `evolved-*.json` 文件 = 8 次独立运行
- 每次运行都 `new Neuroevolution(...)`，创建全新的种群管理器
- **不会**自动加载之前任何 evolved 文件的权重

   ```
   早期：-0.4600 0.0568 0.5764  ← 差距大，多样性好
   后期：-0.6400 0.5764 0.5764  ← med=max，种群趋同
   ```
   中位数接近最大值，说明大部分个体都相似，进化失去探索动力

3. **最终水平评估**
   - 适应度 0.58 ≈ **70%胜率**（对 ruleAi）
   - 达到了"**良好**"水平，但仍未到优秀（0.8+）

**改进建议**：
- 30代的最终适应度与5代几乎相同，**说明代数不是瓶颈**
- 尝试增大种群：`POPULATION=50`（增加基因多样性）
- 或怀疑 ruleAi 本身水平有限，换成更强的对手进化
- 调整变异率：`mutationRate=0.2`（默认0.15可能不够）
- 或者，0.58 已经够用，ruleAi 本身就是中等水平对手

---

#### 案例3：深度进化（50个体×30代）与算法上限分析

```
POPULATION=50, GENERATIONS=30
第0代：  bestClean=0.5764 → 起点即峰值！
第1-29代：bestClean=0.5764 → 30代**零提升**
```

**惊人发现**：增大种群到50并没有帮助突破瓶颈，反而第0代就达到了最优值。

### 10.4 算法上限分析与突破策略

通过案例2和案例3的对比，我们发现了当前配置下的**算法上限现象**。

#### 什么情况下达到了上限

**判断标准**：
- 进化10代以上，适应度提升 < 0.01
- 增大种群（20→50）无法突破
- `clean[min med max]` 中 med ≈ max，种群趋同
- 最优个体被交叉变异"破坏"后无法产生更好个体

**你的实验证据**：
| 实验 | 配置 | 最终适应度 | 提升空间 |
|------|------|------------|----------|
| 20个体×5代 | 基础 | 0.57 | 有 |
| 20个体×30代 | 增代数 | 0.58 | **已上限** |
| 50个体×30代 | 增种群 | 0.5764 | **起点即上限** |

#### 为什么会遇到上限

这不是"神经进化算法"本身的问题，而是**当前配置组合**遇到了多重瓶颈：

**1. 对手水平限制（最可能）**
- `ruleAi` 是基于简单棋形规则的AI
- 它本身不是强对手，进化找不到"更高水平"的策略
- **比喻**：跟臭棋篓子下棋，越下越臭，找不到更好的走法

**2. 网络结构太简单**
```
当前：[6, 4, 1] = 28个参数
     6维输入 → 4个隐神经元 → 1个输出
```
- 28个参数可能不足以表达复杂的五子棋策略
- 隐层只有4个神经元，非线性表达能力有限
- 就像一个"小学生大脑"去解大学数学题

**3. 特征设计瓶颈**
- 6维特征（位置、规则分、候选数、进度）是"摘要"
- 不是完整的棋盘编码，信息有损失
- 可能不足以区分复杂局面

**4. 适应度地形崎岖（高原现象）**
```
        0.5764  ← 你在这里（局部最优平台）
           ↓
    [周围都是平地或悬崖]
           ↓
    微小变异 → 掉到 0.4、0.3（变差）
    没有渐进的上坡路可走
```
- 0.5764 不是一个"坡顶"，而是一个"平台"顶端
- 周围没有渐进优化的路径，只有跳跃才能突破

#### 突破上限的解决方案

**方案1：换更强的对手（推荐）**
- 用MCTS（蒙特卡洛树搜索）AI作为对手
- 或者更深层的规则AI
- **预期效果**：可能突破0.58，达到0.7-0.8

**方案2：增大网络结构**
```javascript
// 从 [6, 4, 1] 改为 [6, 8, 1] 或 [6, [8,4], 1]
// 参数从28个增加到 6×8 + 8×1 = 56个 或更多
```
- 更多参数 = 更强的表达能力
- **代价**：进化时间增加，需要更多代数

**方案3：增加输入特征**
- 从6维增加到10+维
- 可以加入：己方最长连子数、对手威胁等级、棋盘中心距离等
- **代价**：需要重新设计特征工程

**方案4：自适应变异策略**
```javascript
// 前期高变异率（探索），后期低变异率（精细搜索）
if (generation < 10) mutationRate = 0.3;  // 大范围搜索
else if (generation < 20) mutationRate = 0.15;  // 正常
else mutationRate = 0.05;  // 精细微调
```

**方案5：自对弈进化**
- NN vs NN 互相进化，而不是 vs ruleAi
- **风险**：可能产生"针对特定弱点"的策略，不够通用

#### 实用建议

**对于当前项目（NN辅助规则AI）**：
- **0.58（70%胜率）已经足够好**
- 作为"辅助"而不是"主导"，这个水平完全够用
- 再提升边际收益很低，投入产出比不高

**如果想追求更强的纯NN五子棋AI**：
- 需要**更大网络 + 更强对手 + 更长时间**的系统工程
- 这是一个独立的、更复杂的项目

**关键认知**：
> 当前配置的上限 ≈ 0.58（70%胜率对 ruleAi）
> 
> 这不是"神经进化不行"，而是"当前配置组合的天花板"
> 
> 就像用自行车去跑赛车比赛，不是自行车的问题，是工具选择的问题

### 10.5 文件字段说明

示例文件结构：
```json
{
  "nnAssistSchemaVersion": 1,
  "nnAssistWeights": {
    "neurons": [6, 4, 1],
    "weights": [28个浮点数...]
  },
  "evolvedAt": "2026-03-23T05:40:14.396Z",
  "bestFitness": 1.0227555555555556,
  "bestFitnessNote": "clean mean (no noise); win=1+speed, loss=-1, draw=DRAW_FITNESS",
  "generations": 1,
  "population": 2
}
```

| 字段 | 含义 |
|------|------|
| `nnAssistSchemaVersion` | 版本号，必须与代码中 `NN_ASSIST_SCHEMA_VERSION` 一致 |
| `nnAssistWeights.neurons` | 网络结构 `[6, 4, 1]` = 输入6 → 隐藏4 → 输出1 |
| `nnAssistWeights.weights` | 所有连接权重，共 **28个** 浮点数（6×4 + 4×1 = 28） |
| `evolvedAt` | 进化完成时间戳 |
| `bestFitness` | 最优适应度值（越大越好） |
| `bestFitnessNote` | 适应度计算方式的说明 |
| `generations` | 本次进化运行的代数 |
| `population` | 每代种群大小 |

### 10.6 weights 数组的含义

28个权重按**层间连接**顺序排列：

| 权重范围 | 连接关系 | 数量 |
|----------|----------|------|
| `weights[0..23]` | 输入层(6) → 隐藏层(4) | 6×4 = **24个** |
| `weights[24..27]` | 隐藏层(4) → 输出层(1) | 4×1 = **4个** |

**每个权重的意义**：
- **正值** = 正相关（一个神经元激活，另一个也倾向于激活）
- **负值** = 负相关（一个神经元激活，另一个倾向于抑制）
- **绝对值越大** = 影响越强

例如：`weights[0] = -0.104` 表示输入层第0个神经元到隐藏层第0个神经元的连接权重为 -0.104，是一个较弱的负向影响。

### 10.7 如何使用 evolved 文件

1. **手动合并到训练仓库**：将 `nnAssistWeights` 对象复制到 `data/ai-training.json` 的对应字段
2. **自动 POST 到训练服务**：设置环境变量 `EVOLVE_POST_PORT=3847`，进化完成后自动推送
3. **比较多次进化结果**：对比不同文件的 `bestFitness`，选择适应度最高的使用
